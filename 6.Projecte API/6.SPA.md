# Introducció a SPA i refactorització del projecte

Fins ara hem treballat amb una estructura molt útil per aprendre:

* una pàgina per a ciutats
* una pàgina per a països
* una pàgina per a províncies

Per exemple:

```text
/ciutats.html
/paisos.html
/provincies.html
```

Cada pàgina feia la seua petició a la API i mostrava les dades amb Alpine.

Esta forma de treballar és bona per començar, perquè ens ha permés entendre:

* com funciona Express
* com fer rutes GET
* com retornar JSON
* com consumir la API amb `fetch()`
* com mostrar dades amb Alpine
* com treballar amb taules relacionades

Però ara el projecte ha crescut. Ja no només volem mostrar dades: també volem preparar el projecte per a crear, modificar i eliminar registres. Per això convé reorganitzar el frontend i passar a una estructura de tipus SPA.

---

## Què és una SPA

SPA significa:

```text
Single Page Application
```

És a dir:

```text
Aplicació d’una sola pàgina
```

Això vol dir que l’aplicació funciona principalment amb un únic fitxer HTML, normalment:

```text
index.html
```

En lloc d’obrir un fitxer HTML diferent per a cada secció, carreguem una pàgina principal i després JavaScript s’encarrega de canviar el contingut que es mostra.

La idea és esta:

```text
index.html
→ zona principal
→ vista de ciutats, països o províncies
```

Per tant:

* la pàgina principal es carrega una vegada
* el menú i l’estructura general es mantenen
* només canvia la part central
* no cal recarregar tota la web cada vegada
* el projecte queda més preparat per a créixer

---

### Necessitat de passar a SPA

Fins ara cada secció era independent:

```text
ciutats.html
paisos.html
provincies.html
```

Cada fitxer tenia:

* el seu HTML complet
* els seus scripts
* la seua càrrega de dades
* la seua taula
* la seua estructura pròpia

Això funciona, però quan afegim més funcionalitat comença a ser menys pràctic.

Per exemple, quan fem operacions CRUD:

* crear ciutat
* modificar ciutat
* eliminar ciutat

necessitarem formularis, botons i canvis de vista. Si mantenim moltes pàgines separades, haurem de repetir estructura i serà més difícil mantindre el projecte ordenat.

Amb una SPA tindrem una pàgina principal i diferents vistes internes. Això ens permet treballar de manera més clara.

---

### Exemple de la millora

### Abans

L’usuari entra en:

```text
/ciutats.html
```

Després vol vore països:

```text
/paisos.html
```

El navegador abandona la pàgina actual, carrega una altra i torna a muntar tota l’estructura.

Després vol vore províncies:

```text
/provincies.html
```

Torna a haver-hi una recàrrega completa.

---

### Ara amb SPA

L’usuari entra en:

```text
/index.html
```

I després tria una secció:

```text
Ciutats
Països
Províncies
```

La pàgina principal continua oberta.

Només canvia la zona central, per exemple:

```html
<div id="app"></div>
```

Això fa que el projecte siga més net, més còmode de mantindre i més paregut a una aplicació real.

---

* **Estructura antiga del projecte frontend**

```text
public/
├── ciutats.html
├── paisos.html
├── provincies.html
└── js/
    ├── ciutats.js
    ├── paisos.js
    └── provincies.js
```

Esta estructura ens ha servit per aprendre perquè cada recurs estava separat en una pàgina pròpia.

El problema és que cada HTML repetix parts comunes:

* `DOCTYPE`
* `<head>`
* CSS
* scripts
* títol
* contenidor principal

A més, si més avant afegim nous recursos, podríem acabar amb massa fitxers HTML:

```text
usuaris.html
productes.html
comandes.html
...
```

---

## Estructura nova que volem

```text
public/
├── index.html
└── js/
    ├── app.js
    ├── ciutats.js
    ├── paisos.js
    └── provincies.js
```

Ara el frontend tindrà una pàgina principal i diversos fitxers JavaScript.

---



### `index.html`

Serà la pàgina principal de l’aplicació.

Contindrà:

* l’estructura general
* el menú
* la càrrega dels scripts comuns
* la zona on es mostraran les diferents vistes

Per exemple:

```html
<div id="app"></div>
```

Esta zona serà la part que canviarà segons la secció activa.

---

### `app.js`

Serà el fitxer central del frontend.

Controlarà:

* l’arrencada inicial
* la navegació interna
* el canvi de vistes
* la coordinació general de l’aplicació

---

### Els `ciutats.js`, `paisos.js` i `provincies.js`

Continuaran existint, però ara només contindran el codi específic de cada recurs.

És a dir, cada fitxer s’encarregarà únicament de la seua part de l’aplicació:

* carregar i mostrar dades
* llistar registres
* preparar formularis
* crear nous registres
* editar registres existents
* eliminar registres

Això permet separar responsabilitats, tindre el codi més ordenat i facilitar el manteniment del projecte.

---

### Millores que ens ofereix la nova estructura

La nova estructura ens ajuda sobretot en tres coses.

* **1. Menys repetició**

Només tindrem un `index.html`.

Això evita repetir el mateix `<head>`, els mateixos scripts i la mateixa estructura general en cada pàgina.

* **2. Millor organització**

Cada part del projecte tindrà el seu lloc:

* la pàgina principal en `index.html`
* la coordinació general en `app.js`
* cada recurs en el seu propi fitxer JS

Això fa que siga més fàcil trobar i modificar el codi.

* **3. Preparació per al CRUD**

Quan afegim POST, PUT i DELETE, serà més senzill treballar dins d’una mateixa aplicació.

Per exemple, en la vista de ciutats podrem tindre:

* taula de ciutats
* botó per afegir ciutat
* botó per modificar
* botó per eliminar
* formulari de ciutat

Tot dins de la mateixa aplicació, sense haver de crear una pàgina HTML nova per a cada acció.

---

## Quins canvis farem

La refactorització consistirà en:

* crear un `index.html` principal
* deixar de dependre de `ciutats.html`, `paisos.html` i `provincies.html`
* crear un fitxer `app.js` per coordinar la navegació
* mantindre fitxers separats per recursos
* fer que les dades es continuen carregant des de la API
* mostrar les vistes dins d’una zona central

No canviem el backend.

Les rutes de l’API continuaran sent les mateixes:

```text
/api/ciutats
/api/paisos
/api/provincies
```

El que canviem és la forma en què el frontend organitza i mostra les dades.

---

# Refactorització del projecte per a convertir-lo en SPA

Ara que ja hem vist què és una SPA, per què la necessitem i com quedarà la nova estructura, anem a fer la transformació del projecte pas a pas.

L’objectiu és passar de:

```text
diversos HTML independents
````

a:

```text
una sola pàgina principal + diferents vistes internes
```

És a dir, deixarem arrere:

```text
ciutats.html
paisos.html
provincies.html
```

i començarem a treballar amb:

```text
index.html
app.js
ciutats.js
paisos.js
provincies.js
```

---

### Situació inicial

Partim d’una estructura pareguda a esta:

```text
public/
├── ciutats.html
├── paisos.html
├── provincies.html
└── js/
    ├── ciutats.js
    ├── paisos.js
    └── provincies.js
```

Cada HTML funcionava per separat.

Ara volem que tot es gestione des d’un únic punt d’entrada:

```text
index.html
```

---

## Pas a pas:

### 1. Crear el nou `index.html`

Dins de la carpeta `public`, crea un fitxer nou:

```text
index.html
```

i copia este contingut:

```html
<!DOCTYPE html>
<html lang="ca">
<head>
    <meta charset="UTF-8">
    <title>Projecte API</title>

    <link rel="stylesheet"
          href="https://cdnjs.cloudflare.com/ajax/libs/picnic/7.1.0/picnic.min.css">

    <script src="js/app.js" defer></script>
    <script src="js/ciutats.js" defer></script>
    <script src="js/paisos.js" defer></script>
    <script src="js/provincies.js" defer></script>

    <script defer
            src="https://cdnjs.cloudflare.com/ajax/libs/alpinejs/3.15.0/cdn.min.js">
    </script>
</head>
<body>

<main class="container">

    <h1>Projecte API</h1>

    <div id="app"></div>

    <hr>

    <div>
        <button onclick="mostrarCiutats()">Ciutats</button>
        <button class="success" onclick="mostrarPaisos()">Països</button>
        <button class="warning" onclick="mostrarProvincies()">Províncies</button>
    </div>

</main>

</body>
</html>
```

Este nou fitxer serà la base de tota l’aplicació.

Conté:

* el títol general
* el menú principal
* la càrrega de tots els JavaScript
* un contenidor central:

```html
<div id="app"></div>
```

Açò és molt important.

A partir d’ara, les diferents vistes es mostraran dins d’esta zona.

---

### 2. Crear `app.js`

Ara crea el fitxer:

```text
public/js/app.js
```

i copia este codi:

```javascript
function mostrarCiutats() {
    document.getElementById("app").innerHTML = vistaCiutats();
    Alpine.initTree(document.getElementById("app"));
}

function mostrarPaisos() {
    document.getElementById("app").innerHTML = vistaPaisos();
    Alpine.initTree(document.getElementById("app"));
}

function mostrarProvincies() {
    document.getElementById("app").innerHTML = vistaProvincies();
    Alpine.initTree(document.getElementById("app"));
}

window.onload = function () {
    mostrarCiutats();
};
```


Este fitxer coordina l’aplicació.

Quan l’usuari prem un botó del menú:

```text
Ciutats
Països
Províncies
```

es canvia el contingut de la pagina central i després s’inicialitza Alpine perquè active les directives dins del nou contingut.

Les funcions `mostrarCiutats()`, `mostrarPaisos()` i `mostrarProvincies()` carreguen una vista diferent dins del contenidor principal:

```html
<div id="app"></div>
````

Per exemple:

```javascript
document.getElementById("app").innerHTML = vistaCiutats();
```

Açò fa dues coses:

* `document.getElementById("app")` busca l’element amb id `app`
* `.innerHTML = ...` substituïx el contingut interior

És a dir, usem **DOM tradicional de JavaScript** per canviar la pantalla visible.

Després:

```javascript
Alpine.initTree(...)
```

indica a Alpine que revise el nou HTML carregat i active directives com:

```text
x-data
x-init
x-for
x-text
```

* `window.onload`

```javascript
window.onload = function () {
    mostrarCiutats();
};
```

Quan la pàgina acaba de carregar-se, mostra automàticament la vista inicial de ciutats.

>NOTA: Es podria fer tot amb Alpine?
>
>Sí. També podríem controlar el canvi de vistes amb Alpine.
>
>Però en este moment preferim esta solució perquè és **més clara i més fàcil d’entendre**:
>
>* `app.js` canvia la vista
>* Alpine controla les dades dins de cada vista


### 3. Refactoritzar `ciutats.js`

Ara `ciutats.js` ja no dependrà de `ciutats.html`.

Ara retornarà una vista interna per a la SPA.

Obri:

```text
public/js/ciutats.js
```

i deixa’l així:

```javascript
function vistaCiutats() {
    return `
        <section x-data="ciutatsApp()" x-init="carregarCiutats()">

            <h2>Llista de ciutats</h2>

            <table class="table">
                <thead>
                    <tr>
                        <th>ID</th>
                        <th>Nom</th>
                        <th>Descripció</th>
                        <th>Població</th>
                        <th>Província</th>
                        <th>Imatge</th>
                    </tr>
                </thead>

                <tbody>
                    <template x-for="ciutat in ciutats" :key="ciutat.ID">
                        <tr>
                            <td x-text="ciutat.ID"></td>
                            <td x-text="ciutat.NOM"></td>
                            <td x-text="ciutat.DESCRIPCIO"></td>
                            <td x-text="ciutat.POBLACIO"></td>
                            <td x-text="ciutat.PROVINCIA_NOM"></td>
                            <td x-text="ciutat.IMATGE"></td>
                        </tr>
                    </template>
                </tbody>
            </table>

        </section>
    `;
}

window.ciutatsApp = function () {
    return {
        ciutats: [],

        carregarCiutats() {
            fetch('/api/ciutats')
                .then(function (resposta) {
                    return resposta.json();
                })
                .then((dades) => {
                    this.ciutats = dades;
                });
        }
    };
};
```

---

>**IMPORTANT** Abans teníem:

```text
ciutats.html + ciutats.js
```

Ara tenim:

```text
ciutats.js
```

que fa dos coses:

1. genera la vista interna
2. carrega les dades amb Alpine

Es a dir:

- Hem refactoritzat `ciutats.js` perquè deixe de dependre del fitxer `ciutats.html` i passe a funcionar dins de la nova SPA. 
- Ara el mateix fitxer conté la funció `vistaCiutats()`, que genera el HTML de la vista i que després es carregarà dins de `#app`
- També manté la part d’Alpine amb `ciutatsApp()`, encarregada de fer el `fetch('/api/ciutats')`, guardar les dades i mostrar-les en la taula. D’esta manera, el recurs de ciutats queda més autònom, ordenat i adaptat a la nova estructura del projecte.



## ANNEX 1

### Canvi de noms del fitxers del frontend (per no confondre entre fitxers del backend i del frontend)

Fins ara, alguns fitxers del backend i del frontend compartien el mateix nom.  

Per exemple:

```text
app.js                 (Express principal)
public/js/app.js       (frontend SPA)

routes/ciutats.js      (rutes backend)
public/js/ciutats.js   (frontend ciutats)

routes/paisos.js       (rutes backend)
public/js/paisos.js    (frontend països)

routes/provincies.js   (rutes backend)
public/js/provincies.js (frontend províncies)

```

> Açò funciona perque estan en carpetes diferents, però pot portar a confusió sobre tot quan el projecte creix.
> Per evitar-ho, canviarem el nom dels fitxers del frontend perquè siga més fàcil distingir cada part del projecte.

---

* **Noms nous dels fitxers**

```text
public/js/app.js          -> public/js/spa.js

public/js/ciutats.js      -> public/js/ciutats-vista.js

public/js/paisos.js       -> public/js/paisos-vista.js

public/js/provincies.js   -> public/js/provincies-vista.js

```

* **Recomanació en WebStorm**

Si utilitzem l’opció **Refactor** > **Rename** de **WebStorm**, el canvi de nom es farà automàticament en totes les referències del projecte.

Això evita errors i ens assegura que no queda cap ruta antiga sense actualitzar.


* **Comprovar index.html**

Webstorm ho ahurà fet automàticament, peró per si de cas comprovem que en el fitxer `index.html` s’han actualitzat les rutes dels scripts:

```html
<!-- index.html -->
<script src="js/spa.js" defer></script>
<script src="js/ciutats-vista.js" defer></script>
<script src="js/paisos-vista.js" defer></script>
<script src="js/provincies-vista.js" defer></script>
``` 


* **Canviar el id del contenidor principal (NO ES OBLIGATÒRI)**

Per evitar confusions, canviem en el fitxer `index.html` el lloc on es carrega el frontend, de `app` a `spa`:

```html
 <div id="app"></div>
 ```

 per

```html
<div id="spa"></div>
```

No és necessari este canvi, peró si recomanable per evitar confusions. Si ho fem hem de actualitzar el fitxer `spa.js` per a que busque el nou id:

```javascript
// spa.js
function mostrarCiutats() {
    document.getElementById("spa").innerHTML = vistaCiutats();
    Alpine.initTree(document.getElementById("spa"));
};
```    

I ho hem de fer també en les altres funcions `mostrarPaisos` i `mostrarProvincies`.


**Es resum:** 


* **En el fitxer `"index.html"` haurem de canviar el `id` del `div` principal de `"app"` a `"spa"`**
* **En el fitxer `"spa.js"` haurem de canviar totes les referències a `"app"` per `"spa"`**
* **La nostra estructura final del frontend serà:**

```text
public/
├── index.html
└── js/
    ├── spa.js
    ├── ciutats-vista.js
    ├── paisos-vista.js
    └── provincies-vista.js
```
