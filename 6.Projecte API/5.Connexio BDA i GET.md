# Connexió a la base de dades i rutes GET

En esta segona part anem a fer el pas important del projecte:

```text
Express → SQLite → JSON real
```

Ara ja no retornarem dades escrites a mà, sinó dades reals de la base de dades.

---


Abans de connectar la base de dades, necessitem entendre una peça fonamental del projecte: **Knex**.

### Què és Knex

**Knex** és una llibreria de Node.js que ens permet treballar amb bases de dades utilitzant JavaScript.

És el que es coneix com un **query builder**.

Això vol dir que:

* no escrivim SQL directament (o no sempre)
* construïm les consultes amb JavaScript
* Knex genera el SQL per nosaltres

---

### Per a què serveix

Knex ens permet:

* connectar amb la base de dades
* fer consultes (`SELECT`)
* inserir dades (`INSERT`)
* actualitzar (`UPDATE`)
* eliminar (`DELETE`)

Per exemple, en SQL escriuríem:

```sql
SELECT * FROM PAIS;
```

Amb Knex escrivim:

```javascript
db('PAIS').select('*')
```

És més fàcil d’integrar dins del codi JavaScript, i ens proporciona una serie d'avantatges:

* evita escriure molt SQL
* és més llegible per a l’alumnat
* funciona amb diferents bases de dades (SQLite, MySQL, PostgreSQL…)
* s’integra perfectament amb Express

---

### Instal·lar Knex i SQLite

Com que el projecte ja s’ha creat com a projecte Express en WebStorm, **Express ja està instal·lat**.

Ara només necessitem afegir els mòduls nous per treballar amb la base de dades.

Obri la terminal de WebStorm i executa:

```bash
npm install knex sqlite3
```

---

On `knex` és la llibreria que utilitzarem per fer consultes.

i `sqlite3` és el driver que permet a Node.js connectar amb bases de dades SQLite.

> **Nota**: Knex necessita este driver per funcionar amb SQLite.

---

Per conprovar que esta instal·lat:

Obri el fitxer:

```text
package.json
```

Hauries de veure alguna cosa semblant a:

```json
"dependencies": {
  "express": "...",
  "knex": "...",
  "sqlite3": "..."
}
```

## Creació el fitxer de connexió a la base de dades

En l’arrel del projecte (on està `app.js`), crea un nou fitxer:

```text
db.js
```

amb el següent contingut.

```javascript
const knex = require("knex");

const db = knex({
    client: "sqlite3",
    connection: {
        filename: "./ciutats.sqlite"
    },
    useNullAsDefault: true
});

module.exports = db;
```

Guarda el fitxer.

---

En el codi anterior del fitxer `db.js` l'usem per

**1.-  Carregar Knex**

```javascript
const knex = require("knex");
```

Carreguem la llibreria que ens permet fer consultes a la base de dades.

---

**2.- Configurar la connexió**

```javascript
const db = knex({
    client: "sqlite3",
    connection: {
        filename: "./ciutats.sqlite"
    },
    useNullAsDefault: true
});
```

Ací estem indicant:

* el motor de base de dades → `sqlite3`
* el fitxer → `ciutats.sqlite`
* configuració necessària per a SQLite

> **Important**: el fitxer `ciutats.sqlite` ha d’estar en l’arrel del projecte.
> És a dir, al mateix nivell que `app.js`, `db.js` i `package.json`.
> Si no està ahí, la connexió no funcionarà.


---

**3.- Exportar la connexió**

```javascript
module.exports = db;
```

Açò permet utilitzar `db` en altres fitxers, com per exemple en les rutes.

### Importar la connexió a la base de dades

Per poder utilitzar la base de dades dins de les rutes, hem d’importar la connexió creada en `db.js`.

Afig esta línia al principi de `routes/index.js`, just després de crear `router`:

```javascript
const db = require('../db');

```

Sense esta línia, quan intentem utilitzar db(...), apareixerà l’error:

```text
db is not defined
```


---

### Ruta GET – tots els països

Afegix esta ruta dins de `routes/index.js`:

```javascript
router.get('/api/paisos', function(req, res) {
  db('PAIS')
    .select('*')
    .then(function(rows) {
      res.json(rows);
    })
    .catch(function(error) {
      res.status(500).json({ error: 'Error en la base de dades' });
    });
});
```

---

## Important: control d’excepcions en rutes GET

Quan treballem amb la base de dades, les consultes poden fallar per diferents motius:

* error en la connexió
* nom de taula incorrecte
* problema intern de SQLite

Per això, és important que totes les rutes que accedixen a la base de dades tinguen **control d’excepcions**.

Especialment en les rutes amb `:id`, perquè depenen de dades concretes i poden fallar amb més facilitat.

---

### Exemple sense control d’errors

```javascript
router.get('/api/provincies', function(req, res) {
  db('PROVINCIA')
    .select('*')
    .then(function(rows) {
      res.json(rows);
    });
});
```

En este cas:

* si tot funciona, retorna les dades
* si hi ha un error, el servidor pot fallar o no respondre correctament

---

### Exemple amb control d’errors

```javascript
router.get('/api/provincies', function(req, res) {
  db('PROVINCIA')
    .select('*')
    .then(function(rows) {
      res.json(rows);
    })
    .catch(function(error) {
      res.status(500).json({ error: 'Error en la base de dades' });
    });
});
```

Ara:

* si tot funciona, retorna les dades
* si hi ha un error, el servidor respon amb un missatge controlat

---

### Important: control de registre en rutes amb `:id`

Quan treballem amb rutes que utilitzen un identificador (`:id`), pot passar una situació diferent a un error:

* la consulta funciona correctament
* però **no existix cap registre amb eixe ID**

En este cas, la base de dades no dona error, però retorna:

```javascript
null
```

---

### Exemple sense control de registre

```javascript
router.get('/api/provincies/:id', function(req, res) {
  db('PROVINCIA')
    .where('ID', req.params.id)
    .first()
    .then(function(row) {
      res.json(row);
    })
    .catch(function(error) {
      res.status(500).json({ error: 'Error en la base de dades' });
    });
});
```

En este cas:

* si l’ID existix, retorna la província
* si no existix, retorna `null`

---

### Exemple amb control de registre

```javascript
router.get('/api/provincies/:id', function(req, res) {
  db('PROVINCIA')
    .where('ID', req.params.id)
    .first()
    .then(function(row) {
      if (!row) {
        res.status(404).json({ error: 'No trobat' });
      } else {
        res.json(row);
      }
    })
    .catch(function(error) {
      res.status(500).json({ error: 'Error en la base de dades' });
    });
});
```

Ara:

* si l’ID existix, retorna la dada
* si no existix, retorna un missatge clar
* si hi ha un error tècnic, també es controla

---

Hem de tindre en compte que:

* **`.catch(...)` controla errors tècnics**
* **`if (!row)` controla dades inexistents**

Açò evita fallades inesperades i fa que la API siga més robusta i fàcil de depurar.
