# Exercici 1.- Equip de Bàsquet

## Versió 1 – Agrupació directa per posicions (objecte amb arrays)

En esta versió, els jugadors estan agrupats directament dins d’un objecte que representa cada posició.

```json
{
  "nomEquip": "Vall d'Uixo Basket",
  "colorEquipacio": "Blau i blanc",
  "estadi": "Pavello Municipal",
  "entrenador": "Jaume Aragó",
  "plantillaHistorica": false,
  "web": null,
  "jugadorsPerPosicio": {
    "base": [
      { "nom": "Marc Ferrer", "dorsal": 4 },
      { "nom": "Adria Soler", "dorsal": 9 }
    ],
    "alero": [
      { "nom": "Pau Navarro", "dorsal": 7 },
      { "nom": "Sergi Vidal", "dorsal": 12 }
    ],
    "pivot": [
      { "nom": "Hector Gimeno", "dorsal": 15 },
      { "nom": "Ivan Torres", "dorsal": 21 }
    ]
  }
}
```

### Avantatges

* Molt clara visualment.
* Ideal si sempre volem consultar jugadors per posició.
* Fàcil de recórrer en una web per blocs (Bases, Aleros, etc.).

### Inconvenients

* Si un jugador canvia de posició, cal moure’l d’array.
* No és tan flexible si volem afegir més informació a cada jugador.
* No és tan habitual en APIs REST modernes.

---

## Versió 2 – Llista única de jugadors amb camp “posicio”

En esta versió, tots els jugadors estan dins d’un únic array i cada jugador indica la seua posició.

```json
{
  "equip": {
    "nom": "Vall d'Uixo Basket",
    "colorEquipacio": "Blau i blanc",
    "estadi": "Pavello Municipal",
    "entrenador": "Jaume Aragó",
    "plantillaHistorica": false,
    "web": null
  },
  "jugadors": [
    { "nom": "Marc Ferrer", "dorsal": 4, "posicio": "Base" },
    { "nom": "Adria Soler", "dorsal": 9, "posicio": "Base" },

    { "nom": "Pau Navarro", "dorsal": 7, "posicio": "Alero" },
    { "nom": "Sergi Vidal", "dorsal": 12, "posicio": "Alero" },

    { "nom": "Hector Gimeno", "dorsal": 15, "posicio": "Pivot" },
    { "nom": "Ivan Torres", "dorsal": 21, "posicio": "Pivot" }
  ]
}
```

### Avantatges

* Model molt habitual en APIs.
* Més flexible si volem afegir camps (altura, nacionalitat, estadístiques…).
* Permet filtrar per posició fàcilment en backend o frontend.

### Inconvenients

* No estan agrupats visualment.
* Si volem mostrar-los per posició, cal filtrar prèviament.

---

## Versió 3 – Model mixt amb catàleg de posicions i plantilla estructurada

En esta versió, definim primer les posicions com a catàleg i després agrupem la plantilla segons codis.

```json
{
  "equip": {
    "nom": "Vall d'Uixo Basket",
    "colorEquipacio": "Blau i blanc",
    "estadi": "Pavello Municipal",
    "entrenador": "Jaume Aragó",
    "plantillaHistorica": false,
    "web": null
  },
  "posicions": [
    { "codi": "BASE", "nom": "Base" },
    { "codi": "ALERO", "nom": "Alero" },
    { "codi": "PIVOT", "nom": "Pivot" }
  ],
  "plantilla": {
    "BASE": [
      { "nom": "Marc Ferrer", "dorsal": 4 },
      { "nom": "Adria Soler", "dorsal": 9 }
    ],
    "ALERO": [
      { "nom": "Pau Navarro", "dorsal": 7 },
      { "nom": "Sergi Vidal", "dorsal": 12 }
    ],
    "PIVOT": [
      { "nom": "Hector Gimeno", "dorsal": 15 },
      { "nom": "Ivan Torres", "dorsal": 21 }
    ]
  }
}
```

### Avantatges

* Molt escalable.
* Permet validar fàcilment les posicions.
* Separació clara entre metadades (posicions) i dades (jugadors).
* Ideal per a futurs JSON Schema.

### Inconvenients

* Més complexa.
* Pot ser excessiva per a un projecte molt simple.




# Exercici 2.- Gestió d’Informació Empresarial 

Hem de representar **empleats, projectes i departaments**, i a més que el model permeta **relacionar empleats i projectes**. Les dates van com a **string**.

* Evita duplicar informació pesada (per exemple dades personals dins de projectes).

* Per mostrar informació completa (nom d’empleat dins d’un projecte), cal fer “join” (buscar per codi).
* Hi ha duplicació de relacions (empleat->projectes i projecte->empleats), i això obliga a mantindre coherència.

# Exercici 3.- Transformació XML → JSON 

Hem de transformar l’XML a JSON mantenint l’estructura jeràrquica, amb **productes dins d’un array**, i amb **números com a números** i **booleans reals**.

## Versió 1 – Conversió directa i fidel a l’XML (la més habitual)

En esta versió, fem una clau arrel `productos` i dins un array `producto` (o `productos` directament). Manté molt bé l’estructura.

```json
{
  "productos": [
    {
      "nombre": "Camiseta",
      "precio": 19.99,
      "disponible": true,
      "detalles": {
        "color": "Rojo",
        "talla": "M"
      }
    },
    {
      "nombre": "Pantalón",
      "precio": 29.99,
      "disponible": false,
      "detalles": {
        "color": "Azul",
        "talla": "L"
      }
    }
  ]
}
```

### Avantatges

* Conversió molt clara i directa.
* Manté la jerarquia tal qual estava a l’XML.
* És la versió més fàcil de validar i d’entendre per a alumnat.

### Inconvenients

* Si després volem metadades (data d’exportació, origen, etc.), cal afegir-les a l’arrel.

---

## Versió 2 – Arrel autodescriptiva amb metadades (més escalable)

En esta versió, encapsulem els productes dins d’un objecte amb una clau més informativa i afegim metadades. Els productes continuen dins d’un array.

```json
{
  "cataleg": {
    "tipus": "productos",
    "total": 2,
    "productos": [
      {
        "nombre": "Camiseta",
        "precio": 19.99,
        "disponible": true,
        "detalles": {
          "color": "Rojo",
          "talla": "M"
        }
      },
      {
        "nombre": "Pantalón",
        "precio": 29.99,
        "disponible": false,
        "detalles": {
          "color": "Azul",
          "talla": "L"
        }
      }
    ]
  }
}
```

### Avantatges

* Permet afegir informació extra sense tocar el format dels productes.
* És útil per APIs o fitxers d’intercanvi on convé tindre context.
* Pots afegir fàcilment camps com `origen`, `dataGeneracio`, etc.

### Inconvenients

* És un poc més llarg i pot ser “massa” si només volem la conversió simple.

---

## Versió 3 – Model normalitzat (preparat per a creixement i identificadors)

En esta versió, afegim un `id` als productes perquè siga més fàcil referenciar-los després (encara que l’XML original no en tinga). Manté jerarquia i tipus correctes.

```json
{
  "productos": [
    {
      "id": 1,
      "nombre": "Camiseta",
      "precio": 19.99,
      "disponible": true,
      "detalles": {
        "color": "Rojo",
        "talla": "M"
      }
    },
    {
      "id": 2,
      "nombre": "Pantalón",
      "precio": 29.99,
      "disponible": false,
      "detalles": {
        "color": "Azul",
        "talla": "L"
      }
    }
  ]
}
```

### Avantatges

* Ideal si després es vol guardar en base de dades o fer relacions.
* Facilita actualitzacions (per exemple, “modifica el producte id 2”).
* Escalable si el catàleg creix.

### Inconvenients

* L’`id` no existia en l’XML original (és una decisió de disseny).
* Si l’objectiu és una conversió 100% literal, esta versió se separa un poc del text original.
