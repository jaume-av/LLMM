# Projecte API

En este tutorial anem a preparar **la base del projecte** per treballar amb:

* **Node.js**
* **Express**
* **Knex**
* **SQLite**

L’objectiu és tindre un projecte funcional que:

* arranque un servidor
* es connecte més avant a una base de dades SQLite
* retorne dades en format **JSON** a través d’una **API**

En esta **primera part** anem a:

* crear el projecte en **WebStorm**
* instal·lar **Express**
* crear el servidor
* provar que funciona
* crear la primera ruta API

L’objectiu és que, al final, puguem obrir esta adreça:

```text
http://localhost:3000/api
```

i veure en el navegador:

```text
OK
```

Encara **no treballarem amb la base de dades**. Primer necessitem tindre clara la base del projecte i entendre com funciona Express.

---

### Arquitectura inicial del projecte

En esta fase no anem a fer encara consultes a cap taula.
El que volem és crear una aplicació molt simple que faça açò:

```text
Navegador
   ↓
Petició HTTP
   ↓
Servidor Express
   ↓
Resposta
```

És a dir, volem que el navegador faça una petició a una ruta com `/api` i que el servidor responga correctament.

Per a aconseguir això, necessitem:

* un projecte Node.js
* una llibreria per crear el servidor
* un fitxer principal amb el codi
* una forma de posar el servidor en marxa

Açò és el primer pas abans de començar amb la base de dades.

---

## Tecnologies 

### Node.js

**Node.js** és un entorn que permet executar **JavaScript fora del navegador**.

Normalment JavaScript s’executa en:

* Chrome
* Firefox
* Edge

Amb Node.js podem utilitzar JavaScript per fer coses que abans només associàvem al backend, com per exemple:

* crear servidors
* llegir fitxers
* treballar amb bases de dades
* crear APIs

En este projecte, Node.js serà el motor que executarà el nostre codi.

### Express

**Express** és una llibreria de Node.js que facilita molt la creació d’aplicacions web i APIs.

Sense Express també es podria crear un servidor, però el codi seria més llarg i més complicat. Express ens simplifica molt el treball.

Amb Express podem:

* crear un servidor web
* definir rutes
* gestionar peticions HTTP
* enviar respostes al navegador

Per exemple, amb Express podem escriure una ruta així:

```javascript
app.get("/api", function (req, res) {
    res.send("OK");
});
```

Això significa que, si algú entra a la ruta `/api`, el servidor respondrà amb el text `OK`.

### Knex

**Knex** és una llibreria que ens servirà més avant per treballar amb bases de dades relacionals des de JavaScript.

En esta primera part encara no l’utilitzarem, però sí la instal·larem perquè forme part del projecte des del principi.

### SQLite

**SQLite** serà la base de dades del projecte. També la connectarem més avant.

En esta primera part encara no la necessitem, perquè només volem crear el servidor i provar les primeres rutes.

---

## Què és una API REST

Una **API** és un mecanisme que permet la comunicació entre aplicacions.

En el nostre cas, el servidor oferirà una sèrie de rutes que altres programes o pàgines web podran consultar.

Una API REST sol treballar amb rutes com estes:

```text
/api
/api/alumnes
/api/productes
/api/llibres
```

En esta primera part només volem entendre la idea més simple possible:

* el client fa una petició
* el servidor respon

Més avant, estes respostes seran dades reals en format JSON.

---

### Peticions HTTP

El navegador i el servidor es comuniquen mitjançant **peticions HTTP**.

Els mètodes més habituals són:

* **GET** → obtindre dades
* **POST** → crear dades
* **PUT** → actualitzar dades
* **DELETE** → eliminar dades

En esta primera part només utilitzarem **GET**, perquè només volem fer proves senzilles.

Per exemple, quan escrivim al navegador:

```text
http://localhost:3000/api
```

el navegador està fent una petició **GET** a la ruta `/api`.

##  Creació del projecte  WebStorm

Obri **WebStorm** - **File** - **New Project**.
Si ja tens un projecte obert, ves a:

Ara hem de comprovar que tenim instal·lat el paquet de **Node.js** en WebStorm. Si no, cal instalar-lo.


Es crearà un projecte amb una estructura inicial que ja inclou Express i un servidor bàsic amb:


* un fitxer `app.js` amb el codi del servidor
* un fitxer `package.json` amb les dependències
* una carpeta `node_modules` on s’instal·laran les llibreries
* una estructura de rutes i vistes que més avant podrem modificar
* un fitxer `bin/www` que s’encarrega d’arrancar el servidor
* un fitxer `routes/index.js` que conté la ruta principal
* un fitxer `views/index.pug` que és la vista que es renderitza en la ruta principal

---
L'estructura que genera WebStorm sol ser una estructura semblant a esta:

```text
projecte-api
├── bin
│   └── www
├── node_modules
├── public
│   ├── images
│   ├── javascripts
│   └── stylesheets
├── routes
│   ├── index.js
│   └── users.js
├── views
│   ├── error.pug
│   ├── index.pug
│   └── layout.pug
├── app.js
├── package.json
└── package-lock.json
```

No cal entendre-ho tot ara mateix, però sí convé saber el paper de les parts més importants.

### `app.js`

És el fitxer principal de l’aplicació Express.

Ací es fa la configuració general del projecte:

* es crea l’aplicació
* es configuren middleware
* es carreguen les rutes
* es gestionen errors

És un dels fitxers més importants del projecte.

> Un **middleware** és una **funció** que s’executa entre la **petició** del client i la **resposta** del servidor.

### `bin/www`

És el fitxer que s’encarrega d’arrancar el servidor.

En els projectes Express generats automàticament, el servidor no s’arranca directament des de `app.js`, sinó des d’este fitxer.

Açò és important perquè més avant, quan executem el projecte, no farem:

```bash
node app.js
```

sinó:

```bash
npm start
```

### `routes`

Aci trobem les rutes del projecte.

Les dues que solen aparéixer són:

* `index.js`
* `users.js`

`index.js` gestiona la ruta principal.
`users.js` és només un exemple que crea Express automàticament.

### `views`

Aci estan les vistes del projecte, normalment en **Pug**.

De moment, això ens interessa poc, perquè nosaltres volem fer una API, no una aplicació basada en plantilles del servidor.

### `public`

Esta carpeta conté fitxers estàtics:

* CSS
* JavaScript
* imatges

Més avant ens podrà servir si fem frontend, però ara no la necessitem encara.

```text
http://localhost:3000
```

```text

```

* el projecte s’ha creat bé

### Entendre d’on ix el missatge “Welcome to Express”

Ara anem a veure què està passant realment.

Obri el fitxer:

```text
routes/index.js
```

Veuràs un codi semblant a este:

```javascript
var express = require('express');
var router = express.Router();

router.get('/', function(req, res, next) {
  res.render('index', { title: 'Express' });
});

module.exports = router;
```

Este fitxer defineix la ruta principal del projecte, és a dir, la ruta:

```text
/
```

Quan entrem a:

```text
http://localhost:3000
```

Express executa esta funció:

```javascript
router.get('/', function(req, res, next) {
  res.render('index', { title: 'Express' });
});
```

Açò significa que el servidor **no està enviant un text directament**, sinó que està fent tres coses:

* carrega una vista
* li passa una dada anomenada `title`
* renderitza eixa vista amb **Pug**

Per això apareix la pàgina amb el missatge:

```text
Welcome to Express
```

Però nosaltres no volem començar per una aplicació basada en vistes del servidor.
En esta pràctica volem una **API senzilla**, així que el primer pas serà simplificar esta ruta.

---

### Primer canvi: substituir la vista per una resposta simple

Obri `routes/index.js` i substituïx tot el contingut pel següent codi:

```javascript
var express = require('express');
var router = express.Router();

router.get('/', function(req, res) {
  res.send('Servidor funcionant');
});

module.exports = router;
```

Guarda el fitxer.

Ara anem a entendre què hem fet.

Abans teníem:

```javascript
res.render('index', { title: 'Express' });
```

Ara tenim:

```javascript
res.send('Servidor funcionant');
```

La diferència és important:

* `res.render(...)` carrega una vista
* `res.send(...)` envia una resposta directa

Això ens interessa més perquè simplifica molt el projecte i ens permet centrar-nos en les rutes de l’API.

Si el servidor continua en marxa, para’l amb:

```text
Ctrl + C
```

```bash
npm start
```

Després obri el navegador i entra en:

```text

```

Ara hauries de veure:

```text

```

Perfecte. Ja tenim control sobre la ruta principal i ja no depenem de la vista `index.pug`.

### Crear la primera ruta API

Ara que la ruta principal ja està simplificada, anem a afegir una ruta pròpia d’API.

L’objectiu és que, quan entrem a:

```text
http://localhost:3000/api
```

el servidor responga:

```text
OK
```

Per fer això, torna a obrir `routes/index.js` i deixa’l així:

```javascript
var express = require('express');
var router = express.Router();

router.get('/', function(req, res) {
  res.send('Servidor funcionant');
});

router.get('/api', function(req, res) {
  res.send('OK');
});

module.exports = router;
```

Guarda el fitxer.

---

### Entendre la nova ruta

La part nova és esta:

```javascript
router.get('/api', function(req, res) {
  res.send('OK');
});
```

Açò vol dir:

* quan arribe una petició **GET**
* a la ruta `/api`
* el servidor respondrà amb el text `OK`

Això ja és una primera ruta d’API.

---

### Provar la nova ruta

Torna a executar el projecte, si cal:

```bash
npm start
```

Ara entra en el navegador a:

```text
http://localhost:3000/api
```

La resposta hauria de ser:

```text
OK
```

Si ho veus correctament, significa que la nostra primera API està funcionant.

---

### Afegir una ruta que retorne JSON

Ara anem a fer un pas més.

Fins ara hem retornat només text. Però en una API el més habitual és retornar **JSON**, perquè és el format que després consumiran altres programes o pàgines web.

Torna a modificar `routes/index.js` i deixa’l així:

```javascript
var express = require('express');
var router = express.Router();

router.get('/', function(req, res) {
  res.send('Servidor funcionant');
});

router.get('/api', function(req, res) {
  res.send('OK');
});

router.get('/api/test', function(req, res) {
  res.json({
    missatge: 'API funcionant correctament',
    any: 2026,
    centre: 'IES Benigasló'
  });
});

module.exports = router;
```

Guarda el fitxer.

---

### Què fa `res.json(...)`

En esta nova ruta apareix esta instrucció:

```javascript
res.json({
  missatge: 'API funcionant correctament',
  any: 2026,
  centre: 'IES Benigasló'
});
```

Açò envia una resposta en format **JSON**.

La diferència amb `res.send(...)` és esta:

* `res.send(...)` envia text o HTML
* `res.json(...)` envia dades estructurades en JSON

Açò és molt important perquè, quan més avant connectem la base de dades, el servidor retornarà les dades exactament així.

---

### Provar la ruta JSON

Si el servidor no està en marxa, executa una altra vegada:

```bash
npm start
```

Ara entra en:

```text
http://localhost:3000/api/test
```

La resposta hauria de ser una cosa semblant a esta:

```json
{
  "missatge": "API funcionant correctament",
  "any": 2026,
  "centre": "IES Benigasló"
}
```

Perfecte. Ara ja sabem retornar text i també JSON.

---

### Codi final de `routes/index.js`

Per a deixar esta primera part acabada, el fitxer `routes/index.js` hauria de quedar així:

```javascript
var express = require('express');
var router = express.Router();

router.get('/', function(req, res) {
  res.send('Servidor funcionant');
});

router.get('/api', function(req, res) {
  res.send('OK');
});

router.get('/api/test', function(req, res) {
  res.json({
    missatge: 'API funcionant correctament',
    any: 2026
  });
});

module.exports = router;
```

### En resum

Hem de quedar-nos amb esta idea:

**Express ens permet crear rutes que responen a peticions HTTP.**

I estes rutes poden retornar:

* text
* JSON

Açò és la base de qualsevol API.

En la segona part connectarem el projecte a **SQLite** i farem que estes rutes retornen dades reals de la base de dades.
